BigHeadCT · Hermes · Next.js

Como o BigHeadCT fala com o Hermes — e como construir isso em Next.js

Este documento resume a integração real do Hermes no BigHeadCT, mostra o desenho do fluxo e descreve um caminho prático para implementar a mesma ideia com Next.js usando uma fronteira server-only, route handlers e açõeses de mutação no servidor.

API prepara o profile Worker executa o diálogo Hermes é serviço externo Next.js mantém segredos no servidor
Leitura sincera Hermes já está profundamente integrado, mas como motor de execução, não como UI.
Decisão arquitetural O melhor ponto de entrada em Next.js é um Route Handler server-only, com Server Actions para forms.
Regra de segurança A chave do Hermes nunca deve ir para o browser. O cliente conversa com a sua rota, não com o Hermes direto.

Fluxo detalhado da integração Hermes

O acoplamento não acontece em uma única chamada; ele atravessa governança, filesystem, seleção de executor e contrato de resposta. O resumo correto é este: a API materializa o perfil do agente, o worker decide se Hermes é o motor daquele run, o executor prepara o payload e o cliente valida a resposta antes de qualquer dado voltar para a aplicação.

Como o diálogo funciona hoje, sem atalhos

1. A mutação nasce na governança

A mudança do agente entra pela API do BigHead. O service valida o payload, grava o estado governado e chama o HermesProfileSync quando a integração Hermes está configurada. 

// API
HermesProfileSync.sync_agent(agent_data)
// gera agent_version_id.yaml no diretório do Hermes

2. O profile vira arquivo no filesystem

O profile é escrito em HERMES_PROFILES_DIR usando escrita temporária e rename atômico. Esse detalhe é importante porque evita profile pela metade e mantém o worker lendo sempre algo consistente. 

// hermes_sync.py
temp_file = ...
os.replace(temp_file, final_path)

3. O worker resolve qual executor usar

Quando o run começa, o worker olha a configuração. Se existe RUN_PROVIDER_URL, ele usa o executor HTTP. Se não existir e o provider default for hermes, ele monta o HermesRunExecutor. 

// worker/main.py
if settings.llm_provider_default == "hermes":
    ctx["run_executor"] = HermesRunExecutor(...)

4. O executor monta o contrato de chamada

O executor prepara headers e payload com Authorization, Idempotency-Key, X-Hermes-Profile, X-BigHead-Organization-Id e X-BigHead-Run-Id. 

// HermesRunExecutor -> HermesClient
headers = {
  "Authorization": "Bearer ...",
  "Idempotency-Key": "...",
  "X-Hermes-Profile": agent_version_id,
}

5. Hermes responde e o cliente valida

O HermesClient lê a resposta e exige contrato mínimo: id, model, usage e choices. Sem isso, o run falha de forma controlada. 

// packages/pycore/src/.../hermes.py
if not response["id"] or not response["model"]:
    raise HermesContractError(...)

6. O resultado volta para a aplicação

Depois da validação, o run devolve content, tool_calls e tokens para o banco, para a fila ou para a UI, sem expor a estrutura interna do Hermes. 

// retorno seguro
{
  provider_event_id,
  model,
  input_tokens,
  output_tokens,
  content,
  tool_calls
}

Onde essa documentação se traduz em contrato técnico

O ponto importante não é o nome de um arquivo específico. O que importa é que toda integração parecida com Hermes precisa explicitar, de forma reutilizável, quatro contratos: ambiente, sync de configuração, execução e validação.

1. Contrato de ambiente

Defina quais variáveis vivem só no servidor, quais podem ser públicas e qual é o ponto de entrada do serviço externo. Em qualquer sistema, isso inclui URL, chave, timeout, modelo default, diretório de artefatos e qualquer identificador de tenant ou workspace.

  • URL do provedor externo
  • chave ou token de autenticação
  • timeout e retry policy
  • diretório ou bucket de artefatos de profile

2. Contrato de sincronização

Sempre que um agente, workflow ou perfil muda, gere uma representação canônica desse estado. Pode ser YAML, JSON ou registro em banco, desde que exista uma fonte de verdade única para o executor consumir depois.

  • entrada: agente/versão aprovada
  • saída: profile canônico consumível
  • persistência atômica ou transacional
  • habilitar/desabilitar sem deixar lixo residual

3. Contrato de execução

O worker ou orquestrador precisa escolher o executor certo em tempo de execução, montar payload, incluir headers de rastreabilidade e tratar o provedor como um serviço externo com contrato rígido, nunca como detalhe interno da UI.

  • seleção de provider por configuração
  • headers de idempotência e rastreio
  • payload alinhado ao profile ativo
  • fallback e erro controlado

4. Contrato de validação

Antes de devolver qualquer resposta à interface ou gravar resultados, valide campos mínimos do provedor: identificador, modelo, uso, conteúdo e metadados de ferramentas, além de uma política clara para falhas de contrato.

  • id da execução/provedor
  • modelo usado
  • tokens de entrada/saída
  • conteúdo final e tool calls, se existirem
Em resumo: para aplicar isso em qualquer sistema, documente o contrato do provedor, o contrato de profile, o contrato do executor e o contrato de resposta. Os nomes mudam; a arquitetura não.

Como construir essa integração em Next.js

A forma mais segura e alinhada ao App Router é manter Hermes no servidor: o browser chama sua rota ou sua Server Action, e o servidor faz a ponte HTTP até Hermes.

  1. Defina o contrato de ambiente

    Separe claramente o que é público e o que é segredo. No browser use apenas variáveis NEXT_PUBLIC_*. Para Hermes, mantenha HERMES_API_URL, HERMES_API_KEY e HERMES_PROFILES_DIR exclusivamente no servidor. 

    # .env.local
NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:55321
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=...
HERMES_API_URL=http://localhost:8642
HERMES_API_KEY=...
HERMES_PROFILES_DIR=F:\Projetos\BigHead\.omc\hermes\profiles
HERMES_DEFAULT_MODEL=hermes
HERMES_TIMEOUT_SECONDS=60

  2. Crie um cliente Hermes server-only

    Centralize a chamada HTTP em um módulo que só roda no servidor. Isso evita duplicação de headers, facilita logs e mantém a validação do contrato em um ponto só. 

    // src/server/hermes.ts
import { HermesClient } from "@/lib/hermes-client";

export function createHermesClient(env: NodeJS.ProcessEnv) {
  return new HermesClient(
    env.HERMES_API_URL!,
    env.HERMES_API_KEY!,
    Number(env.HERMES_TIMEOUT_SECONDS ?? 60)
  );
}

  3. Exponha um Route Handler no App Router

    O Route Handler é a fronteira ideal para o browser chamar. O usuário envia o pedido para sua aplicação e sua aplicação chama Hermes do lado do servidor. 

    // app/api/hermes/chat/route.ts
import { NextResponse } from "next/server";
import { createHermesClient } from "@/server/hermes";

export async function POST(request: Request) {
  const body = await request.json();
  const client = createHermesClient(process.env);
  const response = await client.chatCompletion({
    messages: body.messages,
    idempotencyKey: body.idempotencyKey,
    profile: body.profile,
    organizationId: body.organizationId,
    runId: body.runId,
    model: body.model
  });

  return NextResponse.json(response);
}

  4. Use Server Actions quando a origem for um formulário

    Para fluxos guiados por formulário, Server Actions deixam o código mais compacto e mantêm a mutação do lado do servidor. É a melhor escolha quando o usuário está enviando uma ação de negócio e não precisa de uma API pública separada. 

    // app/actions/send-to-hermes.ts
"use server";

import { createHermesClient } from "@/server/hermes";

export async function sendToHermes(payload: {
  messages: Array<{ role: string; content: string }>;
  profile?: string;
}) {
  const client = createHermesClient(process.env);
  return client.chatCompletion({
    messages: payload.messages,
    idempotencyKey: crypto.randomUUID(),
    profile: payload.profile
  });
}

  5. Valide a resposta antes de aceitar

    A integração só fica confiável quando a resposta é validada. Exija id, model, usage e choices antes de devolver qualquer dado ao frontend. 

    // contrato mínimo
if (!response.provider_event_id || !response.model) {
  throw new Error("Resposta Hermes fora do contrato");
}

  6. Mantenha o frontend simples

    O componente cliente deve só renderizar estados, enviar o formulário e mostrar a resposta. O browser não deve conhecer chave, URL interna nem detalhes da execução. 

    // client component
"use client";

async function onSubmit() {
  const res = await fetch("/api/hermes/chat", {
    method: "POST",
    headers: { "content-type": "application/json" },
    body: JSON.stringify(payload)
  });
  setResult(await res.json());
}

  7. Registre observabilidade e erros

    Faça logs de idempotência, profile, run e status. Isso ajuda a separar falha do Hermes, falha de contrato e falha de rede. 

    // exemplo de log
logger.info("Hermes request", {
  profile,
  runId,
  idempotencyKey,
  model
});

Arquitetura recomendada no Next.js

Quando usar Route Handler

  • quando o browser precisa de uma rota previsível;
  • quando você quer separar API de UI com clareza;
  • quando a chamada pode vir de qualquer cliente, não só de um form.

Quando usar Server Action

  • quando a mutação nasce de um formulário;
  • quando o fluxo é interno ao app;
  • quando você quer reduzir boilerplate sem expor segredo algum.
Minha recomendação prática: use Route Handler como fronteira principal e Server Actions como ergonomia para formulários. Isso costuma dar a melhor combinação entre clareza, segurança e manutenção.
